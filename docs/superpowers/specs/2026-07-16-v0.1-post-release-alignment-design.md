# v0.1 发布后对齐方案

- 状态：Implemented
- 日期：2026-07-16
- 范围：PRD 对齐、发布保护、公开文档、M4 跟踪、两个次要工程问题
- 产品基线：已发布并被 Obsidian Community Plugins 收录的 `0.1.5`
- 代码基线：`main` at `898bb01`
- PRD 来源：GitHub Issue #1 与 `docs/prd/v0.1-first-public-release.md`

## 背景

v0.1 已完成构建、GitHub Release 和 Obsidian Community Plugins 收录。M0–M3
的本地阅读、兼容性、安全、性能、打包与安装证据已经建立；真实用户安装、
真实工作文件和重复使用反馈尚未形成可验收的数据集。

发布后审查同时发现四类需要对齐的事项：

1. 当前代码将诊断摘要和兼容性提示统一放在默认关闭的设置后，而原 PRD
   要求兼容性提示始终可见；产品决策是以当前代码行为为准并修订 PRD。
2. 当前暂不发布新版本，因此不提升版本号；发布流程仍需防止未来复用已经
   发布的版本号或覆盖既有资产。
3. M4 不再作为发布前阻断门槛，改为等待自然产生的真实用户反馈；在证据
   充分前不宣称产品验证完成，也不据此启动 v0.2。
4. README、SECURITY 和隐私描述需要反映已发布状态与实际持久化字段；测试
   不应穿透插件私有状态，数据存储模块的名称也应反映其真实职责。

本方案只定义后续实施工作，不在本次文档变更中修改版本、创建标签、发布
资产或操作 GitHub Issue/Milestone。

## 决策摘要

### D1：诊断摘要和兼容性提示保持一个用户开关

`Diagnostic summary` 默认关闭。关闭时：

- 不显示 `unsupported-content` 与 `font-substitution` 兼容性提示；
- 不显示复制诊断摘要入口；
- 插件仍可在当前视图生命周期内完成本地兼容性检查，但不持久化文档内容，
  不上传、不发遥测；
- 阻断错误、重试和外部默认应用兜底不受该开关影响，始终按错误状态显示。

开启后，新打开、重试或重新加载的 PPTX 显示可检测的兼容性提示和复制诊断
摘要入口。当前实现不向已经打开的视图实时推送设置变化，因此 PRD 和文档应
明确“对下一次打开/重试/重新加载生效”，不额外引入设置订阅机制。

这是有意的产品取舍：默认界面更简洁，用户需要主动开启诊断可见性；外部
默认应用仍是始终可见的可信度兜底。未知的 PowerPoint 渲染差异仍可能无法
自动检测，成功预览不代表像素级或语义级完全等价。

### D2：当前开发继续使用 `0.1.5`，发布时强制新版本

当前没有发布动作，因此：

- 不修改 `package.json`、`manifest.json` 或 `versions.json` 的版本；
- 普通构建、测试、开发 ZIP 和无发布上下文的 `release:check` 继续允许运行；
- 不覆盖线上 `0.1.5` Release 或其任何资产。

新增独立的“发布模式检查”。只有准备真正发布时才执行严格规则，并要求：

1. package、manifest、versions 与发布标签版本一致；
2. 发布标签解析到当前待发布提交；
3. 同版本 GitHub Release 不存在；若已存在则停止并要求提升版本号；
4. 发布工作流不得覆盖或替换既有资产；
5. 同一提交的两次构建保持字节一致。

这样既不会因 `main` 当前仍标记 `0.1.5` 阻塞日常 CI，也能在未来发布入口处
阻止“同版本、不同代码”的资产进入公开渠道。

### D3：M4 改为发布后开放式验证

不主动组织封闭 Beta，不加入遥测，也不把下载次数当作真实安装或重复使用。
真实反馈到达时再记录以下证据：

- 插件版本、Obsidian 版本和操作系统；
- 是否安装在真实工作 Vault；
- 真实 PPTX 数量、页数/文件大小的非识别性范围；
- 是否成功打开、主要内容是否可阅读；
- 是否出现阻断错误、是否开启诊断、是否使用外部打开；
- 是否发生源文件修改、安全执行、启动崩溃或卸载阻断；
- 用户是否在后续日期再次使用。

原来的“5 名用户、20 份文件、7 天无 P0/P1”不再是已经完成公开发布的前置
条件，而是形成 v0.2 继续/停止决策所需的建议证据量。证据不足时，M4 保持
开放，v0.2 搜索、页级引用等工作保持未决。

### D4：一个插件数据深模块，而不是多个浅存储模块

当前 `ReadingPositionStore` 同时持有读取位置、Vault 级设置、持久化 schema、
保存队列、归一化和订阅行为。它们共享同一个 Obsidian `loadData/saveData`
适配器与原子快照，拆成多个存储模块会制造多个浅接口和写入协调问题。

因此将模块和文件重命名为 `OfficeViewerDataStore` / 
`office-viewer-data-store.ts`，保留一个外部接口：

- `initialize`、`flush`、`dispose` 管理整个插件数据生命周期；
- `settings` 和设置更新方法管理 Vault 级偏好；
- `resolve`、`record`、`rename`、`delete` 管理阅读位置；
- `subscribeThumbnailRailWidth` 保留跨视图同步。

调用者和测试继续通过这个接口操作，不接触内部 `data`、revision 或保存队列。
此次只做诚实命名和引用迁移，不改 schema version，不拆分适配器，不改变用户
数据，也不引入迁移成本。

## PRD 修订方案

GitHub Issue #1 是 PRD 的权威来源。本地 `docs/prd/v0.1-first-public-release.md`
作为仓库内可审查快照，应增加 Issue 链接和同步日期，并与 Issue 正文保持一致。

### Solution

将“复杂或不支持的内容必须显示降级提示”改为：

> 插件在本地检测已知兼容性风险。诊断摘要默认关闭；用户开启后，新打开、
> 重试或重新加载的文件显示可检测的兼容性提示和诊断复制入口。阻断错误、
> 重试和外部默认应用兜底始终可见。

### Release success criteria

将“对不支持或明显失真的测试文件，100% 能显示错误或兼容性提示”拆为：

- 阻断性不支持、加密、损坏、资源限制和安全拒绝 100% 显示稳定错误与可用
  的外部兜底；
- 对语料库中可检测的非阻断风险，开启诊断摘要后 100% 显示兼容性提示；
- 关闭诊断摘要时不显示非阻断兼容性提示，README 和设置页必须解释该行为。

将真实用户目标从“发布成功条件”移动到“M4 发布后验证信号”，避免已经完成
的市场发布与尚未发生的产品验证互相矛盾。

### User stories

修订以下故事：

- Story 26：用户开启诊断摘要后，希望看到可检测的不支持内容提示；
- Story 28：用户开启诊断摘要后，希望看到可检测的字体替换提示；
- Story 35：设置页解释本地处理、兼容性边界、诊断默认关闭和生效时机；
- Story 37：用户主动开启诊断摘要后，可以复制不包含文档内容的摘要。

Story 24、25、27、32、34 和 38 保持不变：阻断错误、稳定分类、重试和外部
打开不受诊断开关控制。

### Implementation Decisions

替换“兼容性警告必须始终可见、可发现”为：

- 非阻断兼容性提示与诊断复制入口由同一个默认关闭的 Vault 级设置控制；
- 设置变化从下一次打开、重试或重新加载开始生效；
- 阻断错误始终可见；
- 外部默认应用兜底在 ready、degraded 和 error 状态继续可用；
- 诊断开关和值可以持久化，但诊断内容、警告结果和文档内容不持久化。

### Testing Decisions

补充默认与开启两种安装态：

1. 新安装默认关闭，非阻断提示和复制入口均不可见；
2. 测试通过 Obsidian 设置界面开启诊断摘要；
3. 重新打开、重试或重新加载测试文件后，提示和复制入口可见；
4. 无论开关状态，阻断错误、重试、外部打开和源文件不变性均成立；
5. E2E 不通过强制类型转换访问插件私有 store。

### M3

把 M3 的“兼容性警告”交付物明确为“可检测、可选显示的非阻断兼容性提示
和始终可见的稳定阻断错误”。M3 仍保留兼容性语料、视觉回归、内容无关诊断、
隐私安全文档和发布质量证据。

### M4

将里程碑名称调整为“发布后真实反馈与 v0.2 决策”，并改写为：

- 已完成：GitHub Release、Community Plugins 收录、补丁发布能力；
- 进行中：等待真实工作 Vault 和真实 PPTX 的自愿反馈；
- 不采用：自动遥测、下载量替代安装、为关闭里程碑而伪造 Beta 数据；
- 完成条件：证据足以作出书面的 v0.2 继续/停止决定，或维护者明确决定长期
  保持 v0.1 维护模式。

5 名用户、20 份真实文件和 7 天稳定观察保留为建议证据量，不再表述为市场
发布前置门槛。

### Release Acceptance Checklist

更新对应条目：

- “所有已知不支持内容有警告或安全失败”改为“阻断性不支持安全失败；开启
  诊断摘要后，可检测的非阻断风险显示提示”；
- “Beta 行为门槛已满足”从发布验收移到发布后验证清单；
- “Community Plugins 提交就绪”更新为“Community Plugins 已收录”；
- 新增“公开文档与当前发布状态、诊断默认值和持久化字段一致”；
- 新增“发布模式拒绝复用已经发布的版本号或覆盖既有资产”。

## 代码与测试设计

### 1. 数据存储模块重命名

涉及文件：

- `src/reading-position-store.ts` → `src/office-viewer-data-store.ts`；
- `tests/reading-position-store.test.ts` → `tests/office-viewer-data-store.test.ts`；
- `src/main.ts`、`src/office-viewer-setting-tab.ts` 及测试中的导入和类型名。

迁移要求：

- `ReadingPositionStore` 更名为 `OfficeViewerDataStore`；
- `OfficeViewerData`、`OfficeViewerSettings`、`OfficeViewerDataAdapter` 和持久化
  JSON 结构保持不变；
- 不修改 `schemaVersion: 1`；
- 既有 `data.json` 加载、保存、关闭开关和清除语义保持兼容；
- 测试名称从阅读位置单一职责改为完整插件数据职责。

### 2. 设置页 E2E 走用户接口

新增测试辅助流程，只封装可观察的 Obsidian UI 操作：

1. 打开 Settings；
2. 进入 Office Viewer 设置页；
3. 根据可访问名称找到 `Diagnostic summary` toggle；
4. 点击并确认其可观察状态；
5. 关闭 Settings，重新打开测试 PPTX。

替换 `tests/e2e/multilingual.e2e.ts` 与
`tests/e2e/packaged-release.release.e2e.ts` 中对私有 `plugin.store` 的强制类型
转换。测试不得调用 `setDiagnosticSummary` 或读取私有字段。

建议覆盖：

- 英文、简体中文、繁体中文设置名称与 toggle 可访问性；
- 默认关闭时没有 warning banner 与 copy control；
- UI 开启并重新打开后，两者可见；
- 打包安装、覆盖升级后设置仍可读取；
- 卸载流程不因设置页辅助代码改变。

### 3. 发布模式检查

保留现有 `checkRelease()` 的本地元数据检查，并增加一个小的发布上下文接口，
避免普通调用者承担 Git 与远程状态复杂度。例如：

```ts
interface PublishContext {
  releaseTag: string;
  headCommit: string;
  tagCommit: string;
  publishedReleaseExists: boolean;
}
```

严格发布检查只接受一个已经解析好的 `PublishContext` 并返回全部验证错误；Git
命令和 GitHub 查询留在 CLI/workflow 适配器中。这样规则可以单元测试，脚本不
需要在测试中伪造真实仓库或网络。

新增命令建议：

- `npm run release:check`：现有本地/CI 元数据检查，不要求升版；
- `npm run release:check:publish`：标签发布专用严格检查；
- `release-assets.yml`：在构建和上传前执行 publish 检查。

严格失败场景：

- 标签与 manifest 版本不一致；
- 标签提交与 HEAD 不一致；
- 同版本 GitHub Release 已存在；
- 必需发布文件或许可证缺失；
- package、manifest、versions 不一致。

远程检查失败时默认停止发布，不降级为警告。普通 push/PR CI 不执行远程发布
存在性检查，因此当前 `0.1.5` 开发状态继续保持绿色。

## GitHub 跟踪方案

实施阶段完成本地文档与代码审查后，再执行外部状态变更：

1. 更新 GitHub Issue #1 PRD 正文，并同步本地 PRD 快照；
2. Issue #1 保持关闭，增加“已发布；后续验证由 M4 Issue 跟踪”的状态说明；
3. 创建 `[M4] 收集发布后真实反馈并形成 v0.2 决策` Issue；
4. 使用 `ready-for-human` 标签，因为真实反馈和产品决策需要人工判断；
5. Issue 放入现有 `v0.1 M4 — Beta 与公开发布` milestone；
6. 将 milestone 重命名或说明为发布后验证，保持 open；
7. M1、M2、M3 milestone 在对应报告和主分支验证确认后关闭；
8. 不因为下载量、无反馈或时间经过而自动关闭 M4。

M4 Issue 使用聚合记录，不要求用户提交文件。反馈中若含机密文件、路径、文本、
图片或作者信息，应移除并引导使用内容无关的诊断摘要。

## 验收标准

### 产品与 PRD

- PRD、README、SECURITY、PRIVACY 对诊断默认值、提示可见性、生效时机、
  发布状态和持久化字段没有冲突；
- 默认关闭诊断时，非阻断兼容性提示和复制入口不可见；
- 开启后，在下一次打开/重试/重新加载时可见；
- 阻断错误、重试和外部打开始终可见；
- M4 明确是等待真实反馈的发布后验证，不再冒充已完成。

### 工程

- E2E 只通过设置 UI 改变诊断开关；
- 不再通过强制类型转换访问私有 plugin store；
- `OfficeViewerDataStore` 名称覆盖其完整职责，持久化 schema 和既有数据保持
  兼容；
- 普通 CI 不要求当前升版；
- 发布模式拒绝已存在的同版本 Release 和错误的标签提交；
- 全部必需测试与发布包生命周期测试通过；
- 工作树不包含意外生成文件。

### 外部状态

- 不发布新版本，不修改 `0.1.5` 线上资产；
- M4 有一个明确的 `ready-for-human` 跟踪 Issue；
- M1–M3 与 M4 milestone 状态反映真实完成度；
- 没有真实反馈时不伪造数据、不自动推进 v0.2。

## 非目标

- 本轮不提升版本号、不发布 `0.1.6`；
- 不改变渲染器、兼容性检测算法或安全预检规则；
- 不让诊断设置对已打开视图实时生效；
- 不增加遥测、崩溃上传、用户账户或远程反馈服务；
- 不主动招募封闭 Beta；
- 不实现搜索、页级链接、嵌入、伴生笔记或付费功能；
- 不拆分插件数据为多个独立持久化模块；
- 不重写 M0–M3 历史测量数据。

## 风险与控制

| 风险 | 影响 | 控制 |
| --- | --- | --- |
| 默认隐藏非阻断风险可能让用户高估预览可信度 | 用户未开启诊断时看不到可检测的 SVG/字体风险 | README、设置页和外部打开操作明确预览边界；阻断安全错误始终显示 |
| 设置修改不实时更新已打开视图 | 用户可能认为 toggle 没生效 | 设置描述明确“下一次打开/重试/重新加载生效”，E2E 固化行为 |
| 当前开发包仍叫 `0.1.5` | 手工误上传时形成同版本不同资产 | 发布模式检查拒绝既有 Release；工作流禁止覆盖；当前不发布 |
| 存储模块重命名造成遗漏 | 构建或测试导入失败 | 类型检查、全仓 `rg ReadingPositionStore` 清零、完整测试 |
| 被动等待长期没有反馈 | M4 和 v0.2 决策长期未完成 | 保持状态诚实；允许明确选择长期 v0.1 维护模式，不用下载量伪造验证 |
| PRD Issue 与本地快照再次漂移 | 后续实现依据不一致 | Issue 标为权威来源，本地文件记录 canonical URL 与同步日期 |
