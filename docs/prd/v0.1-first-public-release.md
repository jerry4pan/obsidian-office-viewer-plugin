# PRD：Obsidian PPTX Viewer v0.1 首次可用公开版本

- 状态：Released（v0.1.5）；后续验证由 M4 跟踪
- 权威来源：[GitHub Issue #1](https://github.com/jerry4pan/obsidian-office-viewer-plugin/issues/1)
- 发布目标：首个可安装、可日常阅读、可提交 Obsidian Community Plugins 的公开版本
- 产品形态：Obsidian 桌面端社区插件
- 支持格式：`.pptx`
- 商业版本：免费基础版；本版本不包含付费授权
- 同步日期：2026-07-16

## Problem Statement

使用 Obsidian 管理工作知识的专业用户经常收到客户方案、项目汇报、研究材料和培训课件等 PPTX 文件。为了阅读这些材料，他们必须离开 Obsidian，调用 PowerPoint、Keynote 或其他 Office 软件；如果想把材料留在笔记工作流中，还需要先转换为 PDF、反复截图或依靠文件名和人工翻页重新寻找内容。

现有替代方案各有明显成本：外部 Office 软件无法与 Obsidian 阅读上下文连贯结合；转换 PDF 会引入等待、中间文件和版本同步问题；通用 Office 查看插件可能覆盖格式更广，但对 PPTX 阅读质量、速度和错误反馈投入不足。

用户需要一个可信赖的本地 PPTX 阅读入口。它必须比“转换后再看”更直接，在常见商务演示文稿上达到日常阅读可用的还原效果，并在无法正确显示时诚实提示，而不是无声地展示错误内容。

## Solution

提供一个只读、离线优先的 Obsidian 桌面端插件。用户在 Vault 中点击 `.pptx` 文件后，文件直接在 Obsidian 工作区打开，并提供可调整宽度的缩略图栏、翻页、页码跳转、自动适应窗口、全屏、键盘操作、阅读位置恢复和始终可见的外部应用兜底。

插件在本机解析并渲染 PPTX，不默认调用 LibreOffice 转 PDF，不上传文件，不执行宏或嵌入式脚本，也不修改原文件。渲染采用渐进式策略：优先让当前页尽快可读，再处理相邻页和缩略图。插件在本地检测已知兼容性风险。诊断摘要默认关闭；用户开启后，新打开、重试或重新加载的文件显示可检测的兼容性提示和诊断复制入口。阻断错误、重试和外部默认应用兜底始终可见。

v0.1 的职责是建立一个可靠的免费阅读基础，并验证真实用户是否愿意把它安装到工作 Vault 中。全 Vault 搜索、页级链接、单页嵌入、伴生笔记和付费授权属于后续版本，不纳入本次发布范围。

## Release Goals and Success Criteria

### Release goals

1. 用户无需离开 Obsidian 即可打开并阅读常见的本地 PPTX 文件。
2. 插件具备足以公开发布的安装、升级、错误处理、文档、安全和兼容性质量。
3. 渲染技术被封装在稳定边界后，未来可以更换引擎而不重写 Obsidian 交互层。
4. 建立可重复的兼容性、视觉回归和性能基线，为后续高保真迭代提供证据。
5. 通过真实工作 Vault 中的自愿反馈验证安装意愿、实际打开率和重复使用，而不以口头兴趣或下载量代替行为。

### Release success criteria

- 至少 80% 的代表性测试文件被评为“主要内容可阅读，无阻断性错位”；
- 阻断性不支持、加密、损坏、资源限制和安全拒绝 100% 显示稳定错误与可用的外部兜底；
- 对语料库中可检测的非阻断风险，开启诊断摘要后 100% 显示兼容性提示；
- 关闭诊断摘要时不显示非阻断兼容性提示，README 和设置页必须解释该行为；
- 在约定的参考设备上，20 MB 且不超过 50 页的基准文件冷启动至首张幻灯片可读的 p95 不超过 3 秒；
- 已渲染页面之间切换的 p95 不超过 100 毫秒；
- 发布候选版本不存在已知的数据损坏、安全执行、启动崩溃或无法卸载问题；
- Community Plugins 已收录，发布资产、说明、许可证和版本元数据完整且一致。

性能目标是首期工程预算，应在 M0 用真实渲染引擎和参考设备校准；若技术验证证明不现实，必须记录测量数据并重新批准目标，不能静默放宽。

## User Stories

1. As a professional Obsidian user, I want to click a PPTX file in my Vault and open it inside Obsidian, so that I can keep my reading context in one application.
2. As a professional Obsidian user, I want the plugin to open PPTX files without first converting them to PDF, so that I do not wait for or manage intermediate files.
3. As a privacy-conscious user, I want PPTX parsing to happen locally, so that confidential work materials are not uploaded to a third party.
4. As an offline user, I want previously unviewed PPTX files to open without an internet connection, so that the viewer works while travelling or in restricted environments.
5. As a reader, I want the first slide to become readable before the entire deck is rendered, so that large presentations feel responsive.
6. As a reader, I want to see the current slide number and total slide count, so that I know where I am in the deck.
7. As a reader, I want previous and next controls, so that I can move through slides without leaving the document.
8. As a keyboard user, I want arrow, Page Up, and Page Down navigation, so that I can read efficiently without using the mouse.
9. As a reader, I want to jump directly to a valid slide number, so that I can reach a referenced page quickly.
10. As a reader, I want a scrollable thumbnail rail, so that I can recognize and select slides visually.
11. As a reader of a large deck, I want thumbnails to load progressively, so that hundreds of slides do not freeze Obsidian.
12. As a reader, I want the slide to fit the window automatically, so that the entire slide remains visible when the pane changes size.
13. As a reader, I want to drag the thumbnail rail divider, so that I can enlarge previews and locate a slide quickly without introducing a second main-slide zoom mode.
14. As a keyboard user, I want an accessible thumbnail divider with step controls and a reset action, so that precise pointer movement is not required.
15. As a reader, I want to enter and exit a distraction-reduced full-screen presentation view, so that I can focus on the material.
16. As a user with multiple panes, I want the viewer to resize with its Obsidian pane, so that split-screen note-taking remains usable.
17. As a returning reader, I want the plugin to remember the last slide for each file, so that I can resume where I stopped.
18. As a returning reader, I want the saved position to remain valid after Obsidian restarts, so that reading continuity survives sessions.
19. As a user who opens several decks, I want each viewer tab to maintain independent navigation, thumbnail scroll, and full-screen state, so that one deck does not affect another.
20. As a user who closes a deck while it is loading, I want background rendering to stop, so that abandoned work does not consume CPU or memory.
21. As a user who modifies a PPTX externally, I want reopening the file to show the new version, so that stale rendered content does not mislead me.
22. As a user, I want the original PPTX to remain byte-for-byte untouched, so that using the viewer cannot damage work files.
23. As a security-conscious user, I want macros and embedded executable content never to run, so that opening a deck cannot execute active content.
24. As a user with a malformed PPTX, I want an understandable error instead of a blank pane, so that I know the file could not be read.
25. As a user with an encrypted or password-protected PPTX, I want the limitation identified explicitly, so that I know why opening failed.
26. As a user with unsupported content, I want a visible compatibility warning after I enable diagnostic summary, so that I do not mistake an incomplete rendering for the source of truth.
27. As a user whose deck renders poorly, I want an “Open in default application” action, so that I can continue working immediately.
28. As a user whose system lacks a referenced font, I want the plugin to indicate possible font substitution after I enable diagnostic summary, so that layout differences are understandable.
29. As an Obsidian theme user, I want viewer controls to remain legible in light and dark themes, so that the plugin fits my environment.
30. As an accessibility user, I want controls to have accessible names, visible focus, and logical keyboard order, so that the viewer can be operated without precise pointing.
31. As a user, I want loading progress or a loading state, so that I know the application has not frozen.
32. As a user, I want a retry action after a recoverable loading failure, so that transient problems do not require restarting Obsidian.
33. As a user, I want errors from one deck to remain isolated to that view, so that a bad file cannot crash other Obsidian panes.
34. As a user, I want unsupported legacy `.ppt` files to receive a clear explanation, so that I do not assume they are corrupted.
35. As a user, I want plugin settings to explain local processing, current compatibility boundaries, that diagnostic summary is off by default, and when enabling it takes effect, so that I can make an informed installation decision.
36. As a user, I want to disable optional persistent reading history, so that I control what metadata the plugin stores.
37. As a user who enables diagnostic summary, I want a simple way to copy a diagnostic summary without including document contents, so that I can report problems safely.
38. As a maintainer, I want rendering failures to include a stable error category, so that reports can be grouped without collecting private files.
39. As a maintainer, I want a documented compatibility corpus, so that renderer upgrades can be judged against known behavior.
40. As a maintainer, I want release artifacts to be reproducible from a tagged commit, so that published binaries can be traced to source.
41. As a Community Plugins reviewer, I want the plugin manifest, permissions, network behavior, privacy statement, and usage instructions to be explicit, so that the submission can be evaluated efficiently.
42. As a new user, I want the README to show supported features and limitations before installation, so that expectations match the product.

## Implementation Decisions

- Build a desktop-only Obsidian community plugin using TypeScript and the supported Obsidian plugin APIs.
- Register a dedicated PPTX file view so a `.pptx` opened from the Vault is rendered in an Obsidian workspace leaf rather than treated as an unknown binary file.
- Read source files through the Vault abstraction as binary data. The viewing pipeline is strictly read-only and never writes back to the PPTX.
- Introduce one renderer adapter as the boundary between Obsidian integration and the selected PPTX rendering library. The adapter exposes document loading, slide count, slide rendering, thumbnail rendering, text/compatibility metadata, cancellation, and disposal. Product code must not depend directly on library-specific objects.
- Do not choose the production renderer solely from a package demo. M0 compares the strongest local JavaScript candidates against the same compatibility corpus, bundle constraints, license requirements, load-time measurements, rendering accuracy, cancellation behavior, and maintenance health. Record the selection in an ADR.
- Do not use LibreOffice-to-PDF, Microsoft Office automation, cloud viewers, document servers, or network services in the default viewing path.
- Treat the deck as an immutable source plus ephemeral rendered views. Rendering work is cancellable and scoped to the open viewer instance.
- Render the current slide first, then adjacent slides, then visible thumbnails. Use a bounded work queue so opening a large deck cannot schedule all slides simultaneously.
- Virtualize the thumbnail rail and release rendered resources that move sufficiently far outside the active window. Memory use must grow with the active rendering window rather than total slide count wherever the selected renderer permits.
- Maintain an explicit viewer state model covering idle, loading, ready, degraded, unsupported, and error states. Navigation actions are valid only when the document metadata required by that action is available.
- Persist only user settings and per-file reading position in the plugin data store. Identify a file using stable Vault-relative information plus a change fingerprint sufficient to invalidate stale state. Do not persist document text or slide images in v0.1.
- Respond to file rename, deletion, modification, Vault close, view close, and plugin unload events. All listeners, workers, object URLs, DOM resources, and renderer instances must be disposed deterministically.
- Keep the main slide in automatic fit-to-window mode and expose no manual zoom controls. Make the thumbnail rail resizable instead: persist one Vault-wide preferred width (`168px` default, `120px`–`480px`), cap the actual width at 45% of a narrow reading body without overwriting that preference, update layout live while dragging, rerender thumbnails at the final width on release, reset on divider double-click, and support Arrow/Shift+Arrow keyboard steps.
- Keep controls consistent with Obsidian styling and theme variables. Do not inject global CSS that changes unrelated workspace content.
- Never execute macros, scripts, ActiveX, external relationships, or embedded executables. External hyperlinks are not followed automatically.
- Provide stable user-facing failure categories for unsupported legacy format, encrypted/protected document, malformed package, renderer incompatibility, resource exhaustion, cancelled load, and unknown failure.
- 非阻断兼容性提示与诊断复制入口由同一个默认关闭的 Vault 级设置控制；
- 设置变化从下一次打开、重试或重新加载开始生效；
- 阻断错误始终可见；
- 外部默认应用兜底在 ready、degraded 和 error 状态继续可用；
- 诊断开关和值可以持久化，但诊断内容、警告结果和文档内容不持久化。
- Diagnostic export contains plugin version, Obsidian version, operating system, renderer version, file size, slide count when known, timings, warning/error categories, and anonymized feature flags. It excludes file paths, filenames, extracted text, images, author metadata, and document contents by default.
- Ship no telemetry or analytics in v0.1. Beta measurements are collected through explicit tester reporting and aggregate manual records.
- Package releases with the files required by Obsidian Community Plugins and publish immutable versioned GitHub releases. Keep the manifest version, release tag, and compatibility metadata synchronized.
- Document supported format, known limitations, privacy behavior, security model, external-open fallback, diagnostic contents, installation, upgrade, and uninstall behavior before public submission.
- The v0.1 product is free. Do not add license checks, account systems, payment SDKs, cloud services, or Pro feature flags to this release.

## Testing Decisions

- The primary and highest testing seam is the installed plugin running inside an Obsidian/Electron test Vault: given a PPTX fixture and a user action, assert the externally visible viewer state, rendered slide, navigation result, error message, or fallback action. This seam covers the full path from Vault file selection through plugin registration, binary read, renderer adapter, and UI.
- Keep the number of seams small. Add a renderer-contract conformance suite because visual compatibility cannot be diagnosed efficiently only through full Electron tests. Every candidate or replacement renderer must run against the same contract and fixture corpus.
- Tests assert external behavior, not DOM implementation structure, internal method calls, private state fields, or library-specific object shapes.
- The representative corpus must include: common business slides; multiple aspect ratios; theme/master layouts; text and font substitution; images and transparency; basic shapes; tables; charts; SmartArt or equivalent complex drawing; grouped and rotated objects; speaker notes metadata; external links; missing media; malformed ZIP/XML; encrypted documents; and large decks.
- Visual regression compares deterministic screenshots at a fixed viewport, scale, font environment, Obsidian theme, and operating-system image. Each fixture declares allowed differences and whether it is expected to render, degrade with warning, or fail safely.
- Performance tests measure time to metadata, time to first readable slide, thumbnail readiness, cached navigation latency, memory at steady state, cancellation latency, and cleanup after closing a view. Report distributions rather than a single fastest run.
- Integration tests cover opening from the file explorer, reopening after restart, independent state in multiple leaves, file rename, file deletion, external modification, pane resize, theme changes, plugin disable/unload, and opening a second deck while the first is loading.
- Failure tests cover malformed, protected, unsupported, oversized, and partially supported documents. A passing failure test requires a stable user-facing explanation, no cross-pane crash, resource cleanup, and a working fallback where applicable.
- Security tests confirm active content is never executed, external relationships are not fetched automatically, diagnostic output excludes document content, and no network request is made during a normal offline open.
- Accessibility tests cover keyboard-only navigation, focus order, accessible names, visible focus, the thumbnail resize separator, and legibility in light and dark themes.
- Release tests install the exact packaged assets into a clean Vault, validate startup and uninstall, and verify that the package contents and versions match the tagged release.
- Diagnostic-summary E2E coverage uses the Obsidian settings UI only: default-off installs hide non-blocking warnings and the copy control; enabling the toggle and reopening the file makes them visible; blocking errors, retry, external open, and source immutability hold regardless of toggle state.
- There is no existing test prior art in the repository because implementation has not started. M0 must establish the test harness and fixture licensing rules before product code grows around an untested renderer.

## Milestones

### M0 — 技术可行性与质量基线

**目标：** 证明至少一个本地 JavaScript 渲染方案能够支撑产品，并把不可接受的技术风险提前暴露。

**交付物：**

- 最小 Obsidian 插件骨架和可重复的开发/构建命令；
- 合法可提交仓库的代表性 PPTX 兼容性语料；
- 候选渲染器适配原型和统一测量脚本；
- 渲染对比、首屏耗时、内存、包体、许可证和维护状态报告；
- 渲染引擎选择 ADR；
- 基础视觉回归与 renderer-contract 测试。

**退出条件：**

- 至少一个候选引擎能在参考设备上打开语料库的所有预期可支持文件；
- 至少 80% 的代表性文件达到主要内容可阅读；
- 不支持文件能够分类失败而非导致进程崩溃；
- 基准文件首张幻灯片 p95 不超过 3 秒，或提供数据充分的目标调整提案；
- 许可证允许计划中的开源分发和商业 Pro 演进；
- 主要技术风险、暂不支持特性和替代引擎成本已记录。

### M1 — 可安装插件与基础阅读闭环

**目标：** 用户能够从 Vault 打开 PPTX，并完成最基本的连续阅读。

**交付物：**

- PPTX 文件类型注册和专用工作区视图；
- 本地二进制读取与只读渲染；
- 当前页、总页数、上一页、下一页和页码跳转；
- 加载、空状态、基础错误和重试；
- 外部默认应用打开兜底；
- 真实 Obsidian 环境的主链路集成测试。

**退出条件：**

- 新用户可从干净 Vault 安装开发构建并打开支持的 PPTX；
- 导航边界正确，非法页码不会破坏视图；
- 打开、翻页和关闭不会修改原文件；
- 单个坏文件不会影响其他工作区叶子；
- 主测试接缝覆盖成功、降级、错误和外部兜底路径。

### M2 — 日常可用的阅读体验

**目标：** 从“能打开”提升到可以处理真实工作材料。

**交付物：**

- 虚拟化缩略图栏；
- 自动适应窗口与可调整、可持久化的缩略图栏；
- 键盘导航与全屏阅读；
- 多叶子独立状态；
- 阅读位置持久化及关闭开关；
- 渐进渲染、相邻页预取、任务取消和资源释放；
- 明暗主题与基础无障碍适配。

**退出条件：**

- 50 页基准文件打开和导航达到性能预算；
- 100 页压力文件不会因一次性渲染全部页面而长期冻结界面；
- 关闭视图或切换文件后，后台任务能停止且资源回落；
- 重启 Obsidian 后能恢复有效阅读位置；
- 纯键盘可完成打开后的核心阅读操作。

### M3 — 兼容性、诊断与发布质量

**目标：** 让用户知道什么可信、什么不支持，并满足公开分发的工程要求。

**交付物：**

- 兼容性警告和稳定错误分类（可检测、可选显示的非阻断兼容性提示和始终可见的稳定阻断错误）；
- 加密、损坏、旧版 PPT 和资源不足等失败体验；
- 不包含文档内容的诊断摘要；
- 设置页、隐私与安全说明、兼容性说明；
- README、许可证、贡献和问题反馈指引；
- CI 构建、测试、版本一致性检查和发布资产生成；
- 干净 Vault 的打包安装、升级和卸载验证。

**退出条件：**

- 语料库的每个文件都有明确的支持、降级或失败预期；
- 视觉回归无未解释变化；
- 正常打开无网络请求，诊断信息不包含内容或直接标识；
- 发布包可从标签重复构建，必需文件完整且版本一致；
- 不存在 P0/P1 缺陷，已知限制全部公开记录。

### M4 — 发布后真实反馈与 v0.2 决策

**目标：** 等待真实工作 Vault 和真实 PPTX 的自愿反馈，并形成书面的 v0.2 继续/停止决策。

**已完成：**

- GitHub Release 与补丁发布能力；
- Obsidian Community Plugins 收录。

**进行中：**

- 收集真实工作 Vault 中的自愿反馈；
- 记录安装、打开、重复使用、失败类型和主观可读性。

**不采用：**

- 自动遥测、下载量替代安装、为关闭里程碑而伪造 Beta 数据。

**建议证据量（非发布前置门槛）：**

- 至少 5 名目标用户安装到真实工作 Vault；
- 累计验证至少 20 份真实 PPTX；
- 连续 7 天没有新增 P0/P1 缺陷。

**退出条件：**

- 证据足以作出书面的 v0.2 继续/停止决定，或维护者明确决定长期保持 v0.1 维护模式；
- 没有原文件修改、安全执行、启动崩溃或阻断卸载问题；
- 基于真实行为形成 v0.2 是否进入搜索与页级引用的继续/停止决策。

## Out of Scope

- 旧版 `.ppt` 文件解析或转换；
- DOCX、XLSX、ODF、PDF 或其他文档格式；
- 编辑、保存、另存为或修改 PPTX 内容；
- 动画、切换效果、演讲者视图和放映计时；
- 宏、VBA、ActiveX 或任何嵌入式可执行内容；
- 像素级复刻 Microsoft PowerPoint 的所有排版和绘图行为；
- 默认 LibreOffice 转 PDF、Microsoft Office 自动化或操作系统预览处理器；
- 云端渲染、文档服务器、上传、同步或多人协作；
- 移动端和平板端支持；
- 全 Vault 文本索引和搜索；
- 页级深链、单页 Markdown 嵌入、引用回链和伴生笔记；
- 文本批注、手写批注和评论；
- 账号、支付、许可证校验、Pro 功能或企业管理；
- 自动遥测、崩溃上传或文档内容收集；
- AI 摘要、问答、OCR 或内容生成。

## Further Notes

- 产品定位和商业化边界以 `docs/product-positioning.md` 为上位决策基线。本 PRD 只定义达到首次公开发布所需的免费基础能力。
- v0.1 的成功不等于商业模式得到验证。它验证的是安装意愿、真实文件兼容性和重复阅读行为；付费意愿需要后续用页级搜索与引用原型及付费 Beta 单独验证。
- 公开下载量不能作为继续投入的唯一指标。优先观察真实 Vault 安装、成功打开率、七日重复使用、外部兜底比例和兼容性失败分布。
- 在 M0 完成前，不承诺具体渲染库。选择必须同时考虑渲染效果、性能、许可证、包体、维护状态和可替换性。
- 若 M0 没有候选引擎达到最低可读性与性能门槛，应停止进入 M1，重新评估商业 SDK、原生 Office 自动化或放弃高保真本地渲染，而不是继续堆叠 UI。
- 后续研发任务应沿里程碑拆成可独立验收的纵向切片；本 PRD 作为需求来源，不应直接被当作单个无限期实现任务。
